# Story 1.2: UI Primitives + Analytics + Helpers + Tipografia

**Status:** Draft
**Epic:** RFG Landing v1 — Captação via WhatsApp
**Complexity:** L
**Estimated Effort:** 12-16h
**Depends on:** 1.1 (Setup Foundation)
**Blocks:** 1.3, 1.4, 1.5, 1.6, 1.7

---

## User Story

Como **desenvolvedor da RFG Landing**, quero todos os componentes primitives (Button, Card, Badge, Eyebrow, SectionHeader, Container, AccordionItem, SkipLink), o hook GSAP `useScrollReveal`, o `<AnimationsProvider>`, o stack completo de analytics (GA4 + Meta Pixel + Vercel + ConsentBanner + ScrollDepthTracker), o helper `whatsapp.ts` com as 8 mensagens canônicas e o helper `tracking.ts` com os 8 eventos, para que **as Stories 1.3-1.6 montem as 13 seções compondo primitives prontos** sem retrabalho de UI ou de instrumentação.

---

## Context

Esta story implementa a **camada de primitives + infra** que será consumida pelas 4 stories de seções (1.3-1.6). Sem nada renderizado em produção visualmente além de uma página interna `/dev/components` (gated por env, para QA visual). A Story 1.1 já carregou Manrope+Inter e tokens RFG; aqui validamos visualmente todos os componentes do `design-system.md` §7.

Inclui o **helper `whatsapp.ts`** com as 8 chaves canônicas literais da tabela FR-021 (PRD §5.3) — qualquer divergência de mensagem aqui contamina TODAS as stories seguintes. Inclui também o **`<ConsentBanner>` LGPD** (FR-046) com gate para Meta Pixel, e os 8 eventos de tracking (FR-044 + mapeamento Meta Pixel `whatsapp_redirect → Lead` / `cta_click → InitiateCheckout`).

Referências: `architecture.md` §6, ADR-002, ADR-003, ADR-006; `design-system.md` §7-8 e §10; PRD FR-021/022/023, FR-040/041/042/043/044/045/046/047/048/051; QG 9.4.

## Acceptance Criteria

### Primitives UI (`src/components/ui/`)

- [ ] **AC-1:** `<Button>` implementado com variantes `primary | secondary | ghost | link` e tamanhos `sm (36px) | md (44px) | lg (56px)`; primary usa `bg-rfg-gradient-cta + shadow-cta`; secondary usa `border-2 border-rfg-dark`; todos com focus-visible `shadow-focus`, hover/active states e `disabled` conforme `design-system.md` §7.1; `aria-label` aceito como prop opcional (FR-023).
- [ ] **AC-2:** `<Card>` implementado com variantes `default | elevated | featured | quote` conforme `design-system.md` §7.2; `featured` inclui badge "MAIS PROCURADO" posicionável e faixa de gradiente decorativa no topo (`::before` com 4px height) — preparação para FR-011 Caminho 2.
- [ ] **AC-3:** `<Badge>` implementado com variantes `default | brand | accent | success | warning | outline` e tamanhos `sm | md`; `success md` é o badge SUSEP do Hero (FR-003); `brand` é o "MAIS PROCURADO" (FR-011); `outline` é a tríade de credenciais (FR-003).
- [ ] **AC-4:** `<Eyebrow>` (eyebrow padrão das seções — Manrope 600 uppercase tracking 0.12em color rfg-dark) e `<SectionHeader>` (Eyebrow + h2 + lead opcional) conforme `design-system.md` §7.6.
- [ ] **AC-5:** `<Container>` com props `variant="wide" (max 1200px) | "narrow" (max 760px)` e padding responsivo conforme `design-system.md` §7.5; `narrow` é usado em Seções 2/7/11 (storytelling).
- [ ] **AC-6:** `<AccordionItem>` com `aria-expanded`, `aria-controls`, transição de altura 250ms, chevron rotativo, suporte a single-open via prop controlada — preparação para FAQ (Seção 13, FR-015) e Objeções (Seção 12, FR-014); navegável via Tab/Enter/Space (NFR-011).
- [ ] **AC-7:** `<SkipLink href="#conteudo">` componente reutilizável, focus-visible, montado no `layout.tsx` antes do `<main>` — atende **FR-040** definitivo.

### Animações (`src/components/animations/`)

- [ ] **AC-8:** `<AnimationsProvider>` (componente raiz client) registra `ScrollTrigger` uma única vez via `gsap.registerPlugin`; chama `ScrollTrigger.refresh()` em `requestAnimationFrame` pós-mount — mitigação Risco R1 (`architecture.md` §10).
- [ ] **AC-9:** Hook `useScrollReveal({ stagger?: number })` implementado conforme `architecture.md` ADR-003 com `useGSAP` + dynamic import GSAP + `gsap.matchMedia()` cobrindo `isMobile`, `isDesktop` e `reduceMotion` — `prefers-reduced-motion` desativa animação e mantém `opacity: 1` final (NFR-014, FR-032).
- [ ] **AC-10:** Pacotes instalados pinados conforme NFR-026: `gsap@^3.14`, `@gsap/react@^2`. **Framer Motion e Lenis NÃO podem ser adicionados** (CON-015) — verificar `package.json` e abortar se aparecerem.

### Analytics (`src/components/analytics/` + `src/lib/`)

- [ ] **AC-11:** `<GoogleAnalytics gaId={...}>` carrega GA4 via `next/script strategy="afterInteractive"` com `anonymize_ip: true`; **só carrega em produção** via guard `process.env.NEXT_PUBLIC_GA4_ID && process.env.NEXT_PUBLIC_VERCEL_ENV === 'production'` (FR-048, Risco R5).
- [ ] **AC-12:** `<MetaPixel pixelId={...}>` carrega Meta Pixel **apenas após `consent.marketing === true`** persistido em `localStorage.rfg_consent` (FR-042); idem guard de produção (FR-048).
- [ ] **AC-13:** `<VercelAnalytics>` + `<SpeedInsights>` (do pacote `@vercel/analytics` e `@vercel/speed-insights`) integrados em `layout.tsx` (FR-043).
- [ ] **AC-14:** `<ConsentBanner>` LGPD (FR-046) com 3 categorias: Necessários (sempre ativo, não opcional), Analytics (opt-in) e Marketing (opt-in); botões "Aceitar todos" e "Personalizar"; "Personalizar" abre modal com toggles por categoria; estado salvo em `localStorage.rfg_consent` formato `{ analytics: bool, marketing: bool, timestamp: ISO }` com re-validação anual; dispara `consent_granted` ou `consent_declined`.
- [ ] **AC-15:** `<ScrollDepthTracker>` dispara evento `scroll_depth` em 25/50/75/100% **uma vez por sessão** (debounced via IntersectionObserver), conforme FR-045.
- [ ] **AC-16:** `src/lib/tracking.ts` exporta `trackEvent<E extends EventName>(name, params)` cobrindo os 8 eventos: `cta_click | whatsapp_redirect | scroll_depth | section_view | faq_open | partner_logo_click | consent_granted | consent_declined` (FR-044); dispara em GA4 (sempre que carregado), em Meta Pixel **só com consent marketing** (FR-044 mapping: `whatsapp_redirect → fbq('track','Lead')`, `cta_click → fbq('track','InitiateCheckout')`), e em Vercel Analytics; tipagem `EventParams[E]` strict.

### WhatsApp Helper (`src/lib/whatsapp.ts`)

- [ ] **AC-17:** Enum `WhatsAppMessageKey` com **exatamente 8 chaves canônicas**: `diagnostico | essencial | completa | legado | cta_unico | sticky_nav | objecoes | footer` — atende **FR-021** e **FR-022**.
- [ ] **AC-18:** Mapa `MESSAGES: Record<WhatsAppMessageKey, string>` com **as 8 mensagens literais exatas da tabela FR-021** (texto antes do encode, com pontuação e emoji-em-dash exatos):
  - `diagnostico`: `Olá! Quero agendar meu Diagnóstico de Ângulo Morto Patrimonial.`
  - `essencial`: `Olá! Quero proteger minha família — Caminho Proteção Essencial.`
  - `completa`: `Olá! Quero o plano completo para minha família — Caminho Proteção Completa.`
  - `legado`: `Olá! Quero construir um legado — Caminho Legado Familiar.`
  - `cta_unico`: `Olá! Quero fazer meu diagnóstico gratuito.`
  - `sticky_nav`: `Olá! Vim pela landing. Quero falar com a RFG.`
  - `objecoes`: `Olá! Quero falar direto com a RFG sobre meu diagnóstico.`
  - `footer`: `Olá! Vim pelo site da RFG.`
- [ ] **AC-19:** Função `buildWhatsAppUrl(key: WhatsAppMessageKey): string` retorna `https://wa.me/${phone}?text=${encodeURIComponent(message)}` usando `process.env.NEXT_PUBLIC_WHATSAPP_NUMBER` (default `5582982359028`); valida que o número não tem caracteres não-numéricos.
- [ ] **AC-20:** Função `getWhatsAppLinkProps(key)` retorna `{ href, target: '_blank', rel: 'noopener noreferrer', 'aria-label': string }` com `aria-label` contextual ("Falar no WhatsApp sobre o Caminho Proteção Essencial", etc.) — atende **FR-023**.
- [ ] **AC-21:** Testes unit em `tests/unit/whatsapp.test.ts` cobrindo as 8 chaves: validar URL final exatamente igual à coluna "URL final" da tabela FR-021 (incluindo `%C3%A1`, `%C3%A2`, `%E2%80%94`, etc.) — qualquer divergência de URL falha o teste.

## Dev Notes

- **ADR-003 (GSAP lazy):** dynamic import dentro do `useGSAP` é o padrão canonical. NÃO importar GSAP no provider (apenas registrar plugin lá após dynamic import). Validar no `pnpm build` output que GSAP não vaza no chunk shared (NFR-019).
- **CON-015 (animation lib única):** Framer Motion e Lenis explicitamente proibidos. Fail-fast no PR se aparecerem em `package.json`.
- **Risco R5 (preview vs prod):** GA4 e Meta Pixel SOMENTE em produção. Em preview/dev, `trackEvent` deve fazer `console.debug` em vez de chamar gtag/fbq — implementar guard.
- **FR-046 LGPD:** banner é gate hard para Meta Pixel. GA4 carrega com IP anonimizado mesmo sem consent (interpretação conservadora documentada em `architecture.md` §7.1) — mas pode-se trocar para opt-in total se Anderson preferir.
- **FR-021 (mensagens canônicas):** as 8 strings são **lei**. Qualquer typo, espaço extra, dash errado (— vs -) ou acento ausente quebra o teste unit (AC-21). NÃO refatorar para tipo dinâmico — usar literal type para garantir.
- **`aria-label` por chave:** `diagnostico → "Falar no WhatsApp para fazer o Diagnóstico de Ângulo Morto Patrimonial"`, `essencial → "Falar no WhatsApp sobre o Caminho Proteção Essencial"`, etc. Padronizar e exportar do helper.
- **AccordionItem (single-open):** o componente em si é controlado externamente. A lógica single-open vai no consumidor (FaqSection na 1.6). Aqui só garantir API limpa.
- **Página dev `/dev/components`:** **nunca pode aparecer em produção** — guard server-side via `notFound()` se `VERCEL_ENV === 'production'`. Útil pra QA/Anderson validarem visualmente antes de partir pra Story 1.3.
